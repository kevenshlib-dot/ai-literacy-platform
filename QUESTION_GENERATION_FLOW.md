## 知识单元选取逻辑

当前系统的知识单元选取逻辑不是智能排序，而是基于素材解析后的顺序做截取：

1. 单知识单元生成
   直接按 `knowledge_unit_id` 查询一个知识单元。
2. 素材批量生成
   按 `material_id` 查询知识单元，按 `chunk_index` 升序排列，取前 `max_units` 个。
3. 素材预览题库
   同样按 `material_id -> chunk_index 升序 -> limit(max_units)` 取知识单元。
4. 题量分配
   如果是按题型分布生成，则采用平均分配：
   `base = 总题数 / 单元数`，`remainder = 总题数 % 单元数`，前 `remainder` 个知识单元多分配 1 道题。

一句话概括：

```text
按素材查知识单元
-> 按 chunk_index 排序
-> 取前 max_units 个
-> 按固定数量或平均分配题量
-> 调用 LLM 生成
```

## 查询知识单元逻辑

这一层可以分成三个问题看：知识单元是怎么生成的、页面如何查询知识单元、出题服务如何查询知识单元。

### 1. 知识单元如何生成

素材解析完成后，系统会先把原始文本按固定规则切块，再把每个文本块保存为一个 `KnowledgeUnit`。

当前切块规则是：

- `chunk_size = 500`
- `overlap = 50`

也就是每约 500 个字符形成一个知识单元，相邻知识单元之间保留约 50 个字符重叠。

解析阶段保存知识单元时，核心字段包括：

- `material_id`
- `title`
- `content`
- `chunk_index`
- `dimension`

其中：

- `chunk_index` 从 `0` 开始递增
- `dimension` 当前直接继承素材的 `category`

这意味着“知识单元顺序”本质上就是素材被切分后的自然顺序。

### 2. 页面查询知识单元逻辑

前端查看某个素材的知识单元列表时，会调用：

```text
GET /api/v1/materials/{material_id}/knowledge-units
```

后端逻辑是：

1. 先查询素材是否存在
2. 再执行知识单元查询
3. 过滤条件：`KnowledgeUnit.material_id == material_id`
4. 排序规则：`order_by(KnowledgeUnit.chunk_index)`
5. 不加 `limit`
6. 返回该素材下全部知识单元

可以概括为：

```text
根据 material_id 查询
-> 只取该素材的知识单元
-> 按 chunk_index 从小到大排序
-> 返回全部结果
```

接口返回的数据里，知识单元内容会被截断展示：

- `content` 最多返回前 200 个字符
- 同时返回 `id / title / chunk_index / dimension`

### 3. 出题服务查询知识单元逻辑

出题相关服务不会做复杂筛选，主要有以下几种查询方式。

#### 3.1 按单个知识单元生成

逻辑：

```text
select(KnowledgeUnit)
where KnowledgeUnit.id == knowledge_unit_id
```

特点：

- 精确查询一个知识单元
- 不涉及排序
- 找不到就直接报错

#### 3.2 按素材批量生成

逻辑：

```text
select(KnowledgeUnit)
where KnowledgeUnit.material_id == material_id
order by KnowledgeUnit.chunk_index
limit max_units
```

特点：

- 先按素材过滤
- 再按 `chunk_index` 升序
- 最后截取前 `max_units` 个知识单元

#### 3.3 按素材预览题库

查询逻辑和批量生成基本一致：

```text
select(KnowledgeUnit)
where KnowledgeUnit.material_id == material_id
order by KnowledgeUnit.chunk_index
limit max_units
```

额外区别是：

- 先检查素材状态必须是 `parsed` 或 `vectorized`
- 只返回预览题目，不直接入库

#### 3.4 建议题型分布

建议题型分布时，也会先查询知识单元，但这里不加 `limit`：

```text
select(KnowledgeUnit)
where KnowledgeUnit.material_id == material_id
order by KnowledgeUnit.chunk_index
```

随后会基于以下信息计算建议值：

- 知识单元总数 `total_units`
- 全部知识单元内容总长度 `total_content_length`
- 知识单元已有难度的平均值

### 4. 当前查询逻辑的关键特点

当前系统的知识单元查询逻辑有几个很明确的特征：

1. 以 `material_id` 为主过滤条件
   大多数场景都是先确定素材，再查该素材下的知识单元。
2. 以 `chunk_index` 作为唯一主排序依据
   当前没有按标签、难度、长度、质量分数做排序。
3. `max_units` 只是在查询后做截断
   它决定“最多取多少个知识单元参与出题”，不是智能筛选参数。
4. 没有语义检索参与
   当前出题时不会先去向量库检索最相关知识单元，而是直接从关系库按顺序取。

### 5. 用一句话概括查询逻辑

```text
知识单元先由素材切块产生
-> 查询时按 material_id 过滤
-> 按 chunk_index 升序排列
-> 某些出题场景再按 max_units 截前 N 个
-> 然后进入出题流程
```

## 关键代码入口

- 单知识单元生成：`app/services/question_service.py` 中 `generate_from_knowledge_unit()`
- 素材批量生成：`app/services/question_service.py` 中 `batch_generate_from_material()`
- 素材预览题库：`app/services/question_service.py` 中 `preview_question_bank_from_material()`
- 自由生成：`app/services/question_service.py` 中 `preview_questions_free()` / `generate_questions_free()`
- LLM 生成主入口：`app/agents/question_agent.py` 中 `generate_questions_via_llm()`
- 预览勾选保存：`app/services/question_service.py` 中 `batch_create_from_raw()`

## 当前逻辑的局限

当前实现可以稳定运行，但知识单元选取和题量分配仍然偏“顺序驱动”，主要有以下局限：

1. 只按 `chunk_index` 顺序取前 `max_units` 个知识单元
   如果素材较长，后半部分的重要知识点可能完全不会参与出题。
2. 没有按知识单元质量做筛选
   当前不会根据知识单元长度、完整性、主题密度、难度、标签质量等因素做排序。
3. 没有按题型适配知识单元
   例如判断题、单选题、简答题更适合的知识单元类型可能不同，但当前没有区分。
4. 平均分配题量较机械
   每个知识单元拿到的题量主要由平均分配决定，不能体现核心章节和次要章节的差异。
5. 缺少去重与覆盖控制
   相邻知识单元内容相近时，可能生成考点重复的题目。

## 优化建议

如果后续要提升题目质量，建议按以下方向优化知识单元选取逻辑。

### 1. 增加知识单元打分机制

在正式生成前，先为每个知识单元计算一个综合分数，例如：

- 内容长度是否适中
- 是否包含明确概念、定义、因果、案例
- 标签数量是否合理
- 难度字段是否完整
- 是否属于素材中的核心章节

可形成类似如下的评分公式：

```text
知识单元得分 = 内容质量分 + 标签质量分 + 难度权重 + 章节权重
```

然后不再简单取前 `max_units`，而是取“得分最高的前 N 个知识单元”。

### 2. 按题型匹配知识单元

不同题型适合不同类型的知识单元：

- 单选题 / 判断题：适合定义明确、知识点清晰的知识单元
- 多选题：适合包含多特征、多条件的知识单元
- 简答题：适合原理解释、案例分析类知识单元
- 情境题 / 论述题：适合包含应用场景、伦理讨论、实践案例的知识单元

这样可以避免“任何知识单元都平均生成所有题型”的粗放模式。

### 3. 增加覆盖率控制

可在选取知识单元时加入“主题覆盖”策略，避免所有题都集中在一个局部主题：

- 同一维度最多选取若干个知识单元
- 同一标签下的知识单元做去重
- 相似度过高的知识单元只保留一个

这样能让最终题库覆盖更完整。

### 4. 增加重要章节权重

如果素材有章节、标题层级、教学目标、重点标记，可以在解析阶段同步写入知识单元元数据，并在出题时提高这些知识单元的优先级。

例如：

```text
重点章节 > 普通章节
总结性知识点 > 例子说明性段落
定义/原理性内容 > 过渡性叙述内容
```

### 5. 增加生成后去重

在 LLM 生成完题目后，可以增加一道题目去重逻辑，例如基于以下维度判重：

- stem 文本相似度
- knowledge_tags 重合度
- source_knowledge_unit_id 是否相同
- 正确答案模式是否重复过多

这样可以减少“换个说法但本质重复”的题目。

## 推荐的后续演进方案

如果要逐步改造，建议按以下顺序推进：

1. 第一阶段
   在知识单元查询后增加基础过滤和排序，不再只按 `chunk_index` 截取。
2. 第二阶段
   引入题型和知识单元的匹配规则，让不同题型从不同类型内容中生成。
3. 第三阶段
   在生成结果后增加重复检测、覆盖率检测和质量打分。
4. 第四阶段
   将“知识单元选取 + 出题 + 去重 + 审核建议”整合为完整的题库构建流水线。
