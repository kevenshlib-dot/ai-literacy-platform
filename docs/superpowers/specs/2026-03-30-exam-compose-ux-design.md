# Exam Compose UX Design

## 背景

当前试卷编排页存在三个直接影响效率的问题：

1. 在候选题池的非第一页加入题目后，右侧列表会重置回第一页，打断连续选题流程。
2. 候选题池只支持单题加入，不支持批量勾选后一次性加入试卷。
3. 左侧当前试卷只支持组内题目拖拽，不能调整整组题型的先后顺序。

这些问题都集中在同一个页面 [ExamCompose.vue](/home/ps/project/ai-literacy-platform/frontend/src/views/ExamCompose.vue)，后端保存接口已经按 `order_num` 接收完整顺序，因此本次主要是前端交互与 payload 生成逻辑收敛。

## 目标

在不重做页面布局的前提下，增强现有双栏编排页，使用户可以：

- 保持当前分页和筛选条件连续加入候选题
- 在候选题池中多选并批量加入试卷
- 调整整组题型的先后顺序，同时保留组内拖拽排序

## 非目标

- 不重构为表格视图
- 不新增后端接口
- 不支持“多题替换一题”或“批量替换”
- 不支持把单题跨题型分组拖拽

## 设计决策

### 1. 保留现有双栏卡片布局

右侧候选题仍使用完整题干卡片，避免退化为仅适合批量操作但不利于阅读的表格。多选能力直接叠加在现有卡片上，而不是切换成另一套“批量模式”。

### 2. 批量操作只做“加入试卷”

`替换当前题` 保持单题语义，避免出现“当前选中左侧一题，但右侧多选多题时替换语义不明确”的冲突。多选状态存在时，用户只能批量加入或清空选择。

### 3. 题型组顺序与组内题目顺序分层管理

最终顺序由两层共同决定：

- 第一层：题型分组顺序
- 第二层：每个分组内的题目顺序

保存时按“组顺序 -> 组内顺序”展开，重新生成最终 `order_num`。这样后端仍然只接收一维有序列表，无需改接口。

### 4. 整组排序采用按钮而非整组拖拽

当前页面已经有单题拖拽。若再引入分组拖拽，会让拖拽语义同时承担“移动单题”和“移动整组”两类动作，误触风险高，也会显著增加实现复杂度。整组排序先采用“上移 / 下移”控制，明确且稳定。

## 页面行为

### 右侧候选题池

- 每张候选题卡片增加复选框
- 面板顶部显示批量操作条：
  - 已选 `N` 题
  - `批量加入试卷`
  - `清空选择`
- 单题加入时：
  - 保持当前页码
  - 保持筛选条件
  - 保持当前候选区上下文
  - 仅将该题从候选列表移除
- 批量加入时：
  - 一次加入所有已勾选且当前未在试卷中的题
  - 清除这些题的勾选状态
  - 保持当前页码和筛选条件
  - 若当前页被加空且不是第一页，则自动回退到上一页

### 左侧当前试卷

- 每个题型分组标题区域新增 `上移 / 下移` 按钮
- 题型分组整体顺序可调整
- 组内题目拖拽保持不变
- 单题仍只能在同组内移动

## 状态设计

页面新增两类核心状态：

- `selectedCandidateIds: string[]`
  - 记录右侧候选题勾选状态
- `groupOrder: string[]`
  - 记录当前题型分组顺序

原有 `items` 继续作为左侧试卷题目明细的单一数据源。

## 数据流

### 候选题刷新

拆分为两类刷新：

- `refreshCandidates()`
  - 用户主动筛选时调用
  - 先将候选分页重置到第 1 页，再请求
- `fetchCandidates()`
  - 保持当前页直接请求
  - 加题、批量加题、替换题后调用

### 试卷保存

保存前的展开顺序：

1. 按 `groupOrder` 取得题型组先后
2. 取每组当前 `items`
3. 按展开后的顺序重新编号 `order_num`
4. 提交到 `/exams/{examId}/composition`

## 错误处理

- 批量加入时跳过已存在于试卷中的题，并给出汇总提示
- 若没有勾选候选题，点击批量加入时提示用户先选择
- 组顺序移动到边界时禁用对应按钮
- 右侧候选页在加入后变空时，自动退页并重新拉取，避免出现“当前页空白但总数仍大于 0”

## 验证方案

- 单题加入后保持当前分页与筛选条件
- 批量加入后左侧数量、题型分组和默认分值符合预期
- 整组上移/下移后保存 payload 中 `order_num` 顺序正确
- 组内拖拽在引入 `groupOrder` 后仍然正常
- 页面构建通过，确保模板和 TypeScript 状态修改没有破坏现有前端编译
